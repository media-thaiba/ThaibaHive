ThaibaHive Ecosystem

Consolidated Architecture Report

July 7, 2026 Final Draft Inputs: Claude · Gemini · ChatGPT

1. Executive Summary

ThaibaHive will evolve from a monolithic Next.js app into a modular ecosystem of independent domain apps sharing common infrastructure. The migration follows a phased, conservative approach — extract shared code first, then split incrementally only when a concrete need justifies each split.

Core Principle: Solve the UX problem first (RBAC + route groups). Solve the organizational problem later (separate deployments) only if justified.

Why Change?

  • Current problem: 800–1000 staff see 22+ navigation items regardless of department. UI is cluttered, bundle is heavy, unrelated features add complexity.
  • Target state: Each staff member sees only 5–6 relevant items. They can discover and launch additional domain-specific apps from a marketplace.

6. Authentication Flow

The Problem with JWT in URLs

The original proposal passed JWT via query string (?token=<jwt>). This is unsafe because tokens appear in browser history, server logs, can leak via Referer headers, and can be accidentally copied/shared.

The Solution: httpOnly Cookie

User logs into Shell → Shell sets httpOnly cookie
Domain=.thaiba.app; Secure; HttpOnly → Clicks marketplace → navigates to child app → Cookie sent with request → authenticated
Security PropertyImplementation
Not in URLsCookie sent via HTTP header only
Not accessible to JavaScriptHttpOnly flag prevents JS access
Only sent over HTTPSSecure flag ensures TLS-only
Works across subdomainsDomain=.thaiba.app covers all subdomains
CSRF protectedSameSite=Lax prevents cross-site requests

8. Phased Implementation Roadmap

Phase 0 Audit & Decouple — 4-6 days

Audit every import to identify cross-department coupling. Refactor tight couplings into shared packages or move into domain boundaries.

Deliverable: A dependency map showing which domains currently depend on each other.

Phase 1 Monorepo + Shared Packages — 4-6 days

Create apps/ and packages/ directories. Extract @thaiba/ui, @thaiba/db, @thaiba/auth, @thaiba/types, @thaiba/config, @thaiba/utils.

Deliverable: ThaibaHive works identically but imports from shared packages.

Phase 2 RBAC Route Groups (Modular Monolith) — 3-5 days

Group routes by domain. Build RBAC: department_id + role determines visible route groups. Navigation renders only permitted domains.

Deliverable: Staff see 5-6 nav items instead of 22. Single deployment.

Phase 3 True Shell — 5-7 days

Strip Attendance, Leaves, Tasks, Reports, Staff Directory out of Shell. Move into apps/hr/. Shell becomes: Auth, Dashboard, Marketplace, Notifications, Search, Profile, Settings.

Deliverable: Parent app is lightweight. HR domain is its own section.

Phase 4 MediaHive as First Independent App — 5-7 days

Extract Media domain into apps/media/. Set up subdomain media.thaiba.app. Cross-domain auth via cookie.

Deliverable: MediaHive deploys independently. Pattern proven.

Phase 5 Expand (Ongoing)

Extract more domains as needed. Layer on: API gateway, event bus, centralized search (Meilisearch), monitoring stack (Sentry, OpenTelemetry).
